# Identity Model

## Overview

Identity in KMBL is not a single object.

It is an evolving system composed of sources, interpretations, and memory.

---

## Identity Sources

Raw inputs provided to KMBL:

- files
- URLs
- text notes
- images
- references
- annotations

These are never treated as final truth. They are material for interpretation and stateful synthesis.

---

## Identity Facets

Structured dimensions extracted from sources:

- voice
- tone
- aesthetic
- themes
- values
- references
- contradictions

Facets may conflict. Conflict is meaningful.

---

## Identity Profile

A continuously updated synthesis of identity:

- summary
- key traits
- dominant patterns
- unresolved questions

This profile is always provisional.

---

## Memory

Two layers:

### Candidate Memory
- proposed insights
- not yet fully trusted

### Accepted Memory
- durable identity signals
- used for future reasoning

---

## Runtime Placement

Identity exists across multiple state layers.

- Sources enter as input material
- Facets are extracted into working graph state
- Profiles are provisional syntheses updated over time
- Candidate memory remains untrusted until accepted
- Accepted memory becomes durable cross-thread memory

Identity may persist, but it is never treated as final.

---

## Rules

- Identity is never final  
- Ambiguity should be preserved when meaningful  
- Contradictions are signals, not errors  
- Identity evolves through accumulation and refinement  

---

## Anti-Patterns

Do not treat identity as:

- a prompt block
- a static bio
- a resolved personality

---

## Role in Execution

Identity is not a required execution stage.

It functions as:

- context for planning
- input for generation
- signal for long-term system behavior

Identity may influence:

- planner decisions
- generator outputs
- evaluator expectations

Identity does not:

- control execution order
- act as a standalone runtime phase
- directly modify system state

It informs the system, but does not orchestrate it.
